关于软件测试的几点反思

发表于 2015-01-11   |   分类于 软件测试   |  

第一个阶段 发现和解决Bug

在这个阶段的基本思路是尽可能发现更多的Bug,然后推动解决验证Bug。这个阶段测试工作主要集中在发现Bug，要做好这个，需要多个方面的努力，比如：

  • 更高效的发现Bug，考验测试设计的能力
  • 发现Bug之后如何清晰的描述，顶级，以及跟进和验证
  • 对业务和架构的理解能力
  • 发现Bug后举一反三的能力
    大家看到的很多经典的测试书籍讲的都是这个阶段的事情，专业与如何更高效的发现缺陷。上面这些东西都是一个业务测试人员的基本功，看似简单，但是做好并不是一件容易的事情。

如果没有其他方面的改变，会导致每次都有那么多的Bug。不过很多时候，我们的测试停留在这个阶段也是因为现状，比如：

  • 开发基本不自测
  • 提测后很多基本的功能都不能正常使用
  • 项目管理比较混乱

目前有不少的团队是处于这样疲于应对的情况，不只是小公司，一些大公司的部分项目也是如此。随着整个研发体系的发展和深入，我们应该有更高的追求。

第二个阶段 质量的管理

在第一个阶段的基础上，如何让开发少产生Bug呢？如果我们只是发现Bug，解决Bug，那么就是一个消耗战，不能形成一个良性的循环，就不能持续优化，工作的长期累积价值就体现不出来。这个阶段的核心思路是对缺陷做分析和考核，并做研发流程中主要问题的梳理和改善

  • 质量数据的统计和分析

    • 外网的Bug情况，包括事故及影响的程度
    • 测试阶段的Bug数，分布（按系统，团队，开发个人），严重程度，Bug的类别维度
    • Bug的横向跨团队和系统的对比，纵向的和历史情况对比
    • 版本发布的情况，代码变更行数的情况

    从这些数据的收集中就能发现很多的问题，比如问题集中在哪里，哪些模块，那些人，哪些类别，以及有没有改善。

  • 问题的追溯和对于开发的考核

    • 编译失败次数的考核
    • 外网师傅和Bug的数量
    • 测试阶段的Bug, 特别是基础功能Bug和严重Bug

  • 对于测试的考核

    • 漏测：绝对数量或者漏测率
    • 版本的工作量和测试效率
    • 发布延期的情况

  • 推动开发自测

    • 统计测试阶段的Bug中，属于开发可自测发现的比例
    • 给出自测的checklist
    • 自动化验收用例

这个阶段除了业务测试的努力，也体现出了QA的价值，这里的QA指质量管理，专注于质量度量和研发流程管理。

第三个阶段 推动全面的质量提升

  • 研发流程梳理，涉及很多方面，比如：

    • 需求不清楚
    • 跨团队的配合问题
    • 代码版本管理
    • 技术方面的评审和大家的理解

  • 提交测试前后座的一些事情

    • 代码的静态扫描
    • code review

  • 测试能力的提升

    • 自动化
    • 辅助手段（代码覆盖率统计）
    • 拓展测试类型（安全测试，性能测试，兼容性测试）

  • 发布环节的质量把控

    • 上线权限控制
    • 灰度
    • 扩容的快速自动化检查

  • 外网的监控

    • 运维层面的监控，如机器，链路，资源使用，主要组件是否正常
    • 业务指标的监控，比如点击率，Bi系统等
    • 集成在产品里面的监控代码
    • 测试层面的自动化监控

  • 外网师傅和问题的收集，跟进和反向推动

  • 用户的问题反馈和满意度
