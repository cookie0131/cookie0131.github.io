但是，对移动端来说，这样的方式是行不通的，因为移动端主要分为安卓和IOS，而这两端出现的问题一般是不一致的，一致的问题主要是数据问题，这时候是需要后台处理的，所以我们测试的时候需要两端都重点测试，而不会出现先着重测试某一端的问题。

分库分表的集中常见形式：水平分库分表

发表于 2015-06-30   |   分类于 架构   |  

水平分库分表与水平分表的思想相同，唯一不同的是将这些拆分出来的表保存在不同的数据库中。

某种意义上来说，有些系统中使用的‘冷热数据分离’也是类似的实践。在高并发场景下和海量数据量的场景下，分库分表能够有效缓解单机和单库的性能瓶颈和压力，突破Io，连接数，硬件资源的瓶颈。

分库分表的集中常见形式：水平分表

发表于 2015-06-21   |   分类于 架构   |  

水平分表也称为横向分表，就是讲表中不同的数据行按照一定规律分不到不同的数据表中，这样来降低单表的数据量，优化查询性能。最常见的方式就是通过主键或者时间等字段进行hash和取模后拆分。比如：
根据id，把1结尾的放一个库，2结尾的放一个库，以此类推。

水平分表能降低单标的数据量，一定程度上可以缓解查询性能瓶颈，但本质上这些表还在一个数据库中，所以库级别还是有io瓶颈，一般不采取这种做法

分库分表的集中常见形式：垂直分库

发表于 2015-06-13   |   分类于 架构   |  

垂直分库的思路就是按照业务模块来划分出不同的数据库表，二不是像早期一样将所有的数据都放在同一个数据库中。比如把一个DB库拆分成 User库，Order库，Product库

系统层面的服务化拆分操作，能够解决业务系统层面的耦合和性能瓶颈，有利于系统的扩展维护。二数据库层面的拆分，道理也是相同的。与服务的治理和降级机制类似，我们也能对不同业务类型的数据进行分级管理，维护，监控，扩展等

数据库往往容易成为应用系统的瓶颈，而数据库本身属于有状态的，相对于web应用服务器，比较难实现横向扩展。数据库的连接资源比较宝贵且单机处理能力也有限，在高并发场景下垂直分库一定程度上能够突破Io,连接数及单机硬件资源的瓶颈，是大型分布式系统中优化数据库架构的重要手段
